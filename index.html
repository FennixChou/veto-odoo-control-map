老賈操控 Odoo 能力總盤點＋不可控項替代方案

參照 API 需求對照矩陣再往前推——Claude Code(老賈)實際能操控什麼、做到什麼程度、碰壁的每一項有什麼替代解｜官方原始碼查證（含 safe_eval 逐行反證）｜2026-07-22（Odoo 18.0 自架 Community）｜如菓工程統籌

一句話回答：Odoo 幾乎所有「營運與內容」功能老賈都能操控，因為整套系統跑在同一組 API 上、任何公開方法都能遠端呼叫。碰壁的不是「功能」而是三類邊界——①核心層擋住的私有方法（走公開 wizard 繞）②Odoo 主動通知老賈（要小繞道）③少數必須人工一次性取得的憑證（金流帳密/DNS/SSL）。每一個都有已查證的可行替代解，沒有死路。

操控分三級：全自動老賈 API 直接做；半自動要走 wizard/自訂小工具/輪詢；需人工一次取得憑證後老賈接管。VETO 九項需求＋日常營運，落在「全自動」的佔絕大多數。

2. 操控的真正邊界（三類，及為什麼）

邊界一：私有方法——核心層的鐵牆

Odoo 核心（odoo/service/model.py 的 get_public_method，前份已反證）擋下所有底線開頭／@api.private 方法的遠端呼叫。這是好事——AI 只能碰官方設計成可對外的公開商業方法（action_confirm/action_post/button_validate 全公開）。遇到私有方法（開發票 _create_invoices、送綠界 _l10n_tw_edi_send）就走它的公開 wizard 包裝——每個都有正確入口，寫在 §1 表上。半自動

邊界二：Odoo 主動通知老賈——要小繞道（本次最深查證）

「發生 X 就自動叫老賈」這件事，Odoo 18.0 沒有原生出站 webhook，而且反證員逐行確認：server action 的 Python 沙箱（safe_eval）硬擋 import、context 裡也沒有任何 HTTP 工具（黑名單封 IMPORT_NAME 位元碼＋dunder 名稱檢查封 __import__，兩條路都死）——所以「server action 直接發 HTTP 打老賈」官方手段做不到。三條替代路，已排序：

路徑可行做法取捨
(c) 老賈端輪詢（預設推薦）✔ 最實際老賈排程（工作排程器＋claude -p）定時跑 xmlrpc search_read 抓新單（domain 依 create_date）零 Odoo 側程式碼；有 1–5 分鐘輪詢延遲
(b) 自訂 module 方法✔ 要即時才用自架加一個小 addon 的公開 model 方法（正常 Python 可 import requests 發 HTTP）→ base.automation 綁 create/write 觸發真正即時零延遲；維護成本＝一顆 module（自架允許）
(a) server action 直發 HTTP✘ 官方手段不可行safe_eval 封死 import 與 HTTP 工具唯一未支援側門（__import__ 已載入模組）脆弱且屬安全弱點，NEVER 用

結論：VETO 日常事件用 (c) 輪詢就夠（訂單進來 1–5 分鐘內老賈就知道）；要「秒級即時」才升級 (b) 自訂 module。

邊界三：必須人工一次性取得的憑證——之後老賈全接管

這類不是「AI 能力不足」，是現實世界要人簽名/驗證的事：向綠界簽約取得 MerchantID/HashKey、金流商帳密、網域 DNS 設定、SSL 憑證核發、API 金鑰索取。查證確認：這些憑證一旦拿到，填入 Odoo（res.config.settings write）、啟用金流商（payment.provider.state 改 enabled，API 可寫）、產對帳單——全部老賈接管。分界線清楚：取得＝人工一次，之後＝全自動。需人工一次

3. 兩個硬傷的替代解（已查證落地細節）

硬傷① 經銷專屬品項可見性

最實際解＝老賈用 API 建一條 record rule（ir.rule）：綁「經銷商」res.groups 群組，用 domain 過濾 product.template——這是 Odoo 原生的權限機制，後台與 portal 都生效，能做到真正「按客戶群隔離商品」。需要「寫」的只是一條 domain，不是開發整包模組，且 ir.rule/res.groups 都是常規模型、老賈能直接 API 建立。前台可再疊多網站（B2B 站＋選單 visibility 設「需登入 connected」）做雙保險。半自動·一次性設定

為什麼原生不行：eCommerce 商品層沒有「按角色/群組可見性」選項（官方文件確認）；website_id 只能一站或全站、無法按客戶群。所以走 record rule。

硬傷② 月結對帳單／帳齡報表

現成報表（Partner Ledger／Aged Receivable／Follow-up）是 Enterprise-only，但Community 自建完全可行、不需 Enterprise、不需在 Odoo 內寫程式：反證確認 account.move.line 有對帳單需要的全部欄位（partner_id／date_maturity 到期日／debit／credit／amount_residual 未沖銷餘額／reconciled／move_id）——老賈用 search_read／read_group 撈出來、依 date_maturity 分帳齡桶，在老賈這端自產對帳單 PDF（版面完全照 VETO 需求做，比 Enterprise 制式報表更貼）。半自動·老賈自建

4. VETO 落地行動

  1. 部署＝自架 Community（呼應前份報告）：授權費 0、含綠界電子發票模組、API 免費、record rule/theme 全自由。老賈用 API key 經 execute_kw 或自建 MCP 操控。
  2. 事件通知先用輪詢 (c)：零 Odoo 側程式碼，訂單/收款進來老賈定時抓。日後要秒級即時再加一顆自訂 module (b)。
  3. 硬傷用 record rule＋自建對帳單解，兩者都是老賈一次性設定/開發，不必買 Enterprise。
  4. 人工項清單交老闆：綠界簽約取金鑰、網域 DNS、SSL——這幾件你/我協作辦一次，之後系統營運全交老賈。
  5. 一個待確認：綠界電子發票模組 l10n_tw_edi_ecpay 的授權版別（前份查證指公開 Community 庫，本次補查未能從私有 Enterprise repo 反面確證）——開發店實測時第一件驗。
